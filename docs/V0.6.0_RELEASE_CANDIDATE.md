## Installation and upgrade posture

A clean installation uses the least-privilege artifact that meets the deployment need, or the governed `full-suite` v0.6.0 artifact when all optional providers are required, as explicitly selected by the operator:

```bash
ARTIFACT=/absolute/path/to/termux-mcp-server
ARTIFACT_SHA256="$(sha256sum "$ARTIFACT" | awk '{print $1}')"
scripts/termux_deploy.sh install \
  --artifact "$ARTIFACT" \
  --version 0.6.0 \
  --sha256 "$ARTIFACT_SHA256"
```

The configuration layout, fixed `mcp_runtime` service name, versioned deployment root, safe-root policy, authentication posture, and Streamable HTTP protocol are unchanged by the version-only metadata transition.

The physically qualified v0.5.1 exact-main candidate may be retained privately as an internal upgrade/recovery baseline while validating v0.6.0. It is not an authoritative public installation source. There is no previous governed public release for an end user to upgrade from.

## Rollback and removal posture

When v0.6.0 is installed as a clean first release, no `previous` target exists. `rollback` must therefore remain unavailable until a later complete release has been installed as an upgrade. Operators can remove a clean installation while preserving configuration:

```bash
scripts/termux_deploy.sh uninstall
```

During internal qualification, an upgrade from the private validated v0.5.1 baseline must prove failed-candidate recovery, successful v0.6.0 activation, failed-rollback recovery, successful rollback, and configuration-preserving uninstall through the canonical validator.

## Known limitations

- Android control beyond exact-stream volume, shell fallback, arbitrary command execution, global process inspection, arbitrary service mutation, package/network mutation, and unrelated high-impact actions remain disabled.
- GET `/mcp` returns HTTP 405 in the default JSON posture; the separately configured SSE posture is finite, replay-bounded, and never broadcasts across streams.
- Sessions are in-memory, bounded, and intentionally lost on restart.
- Directory, file-copy, file-trash, file-write, and volume authorities with the same exact family/key-id/key/principal share replay, clock, and capacity state across independently constructed routers in one process. Grants expire after 60 seconds and lose their process-local replay state on restart; distinct capability codes prevent cross-use, immediate invalidation requires key rotation, and multi-process consumers require an external atomic one-use coordinator.
- Workflow artifacts are retained for 30 days, then expire; they are qualification inputs rather than a durable distribution channel.
- No byte-for-byte reproducible-build claim is made.

## Publication stop conditions

Do not create the tag or GitHub Release if any of the following is true:

- final exact-main CI, Security, or any Android posture is missing or failed;
- the seven final manifests do not identify the same immutable source commit and version;
- downloaded checksums, sizes, ELF identity, feature posture, or embedded version disagree;
- a fresh direct validator-v11/schema-v2 physical report bound to the full-suite digest does not say `releaseEligible:true`;
- the native ARM64 official-Termux exact-artifact report is missing or failed;
- aggregate schema/gate-v3 evidence does not bind the exact full-suite digest/manifest and prove the 17-disabled, four isolated 18-tool, 21-enabled, and four-filesystem-denial contracts;
- the physical-qualification envelope does not bind both evidence-report digests and the distinct workflow/native full-suite lineages without exposing raw device metadata;
- the protected release stage is absent, non-deterministic, built from anything other than the exact qualified bytes, or lacks a closed `staged_not_released` manifest;
- the `release-qualification` environment protections have not been independently verified before dispatch;
- a blocking review thread, merge conflict, newer failed run, or release-document inconsistency remains;
- durable asset names/checksums or the GitHub Release body have not been independently rechecked.

Tagging and publishing remain separate maintainer actions after this release-preparation pull request and the final exact-main evidence are complete.
