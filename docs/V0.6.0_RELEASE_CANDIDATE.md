If separately approved for publication, the durable GitHub Release assets must be:

- `termux-mcp-server-v0.6.0-aarch64-linux-android-default`;
- `termux-mcp-server-v0.6.0-aarch64-linux-android-mcp-runtime`;
- `termux-mcp-server-v0.6.0-aarch64-linux-android-android-battery-status`;

The default binary exposes only coarse health/readiness endpoints. The `mcp-runtime` binary additionally exposes the authenticated MCP 2025-11-25 Streamable HTTP surface and the documented seven-tool allowlist. Battery, volume, and command artifacts each add only their separately runtime-gated read-only surface. The command artifact exposes fixed server diagnostics, not arbitrary execution. Artifact names and manifests must keep all five postures unambiguous.

Final evidence must be generated after the last v0.6.0 qualification change reaches `main`. It must reconcile one immutable SHA across CI, Security, five Android workflow manifests, downloaded binary digests, embedded `termux-mcp-server 0.6.0` output, and native ARM64 official-Termux runtime validation. Physical properties use either a direct candidate observation or the committed v0.5.1 source report plus a passing inheritance report that proves unchanged runtime/dependency/deployment inputs and exact bridge digests. Runtime implementation changes, including the fixed-command lane, invalidate metadata-only inheritance. Values are recorded in the GitHub Release body only after they exist; this repository does not contain guessed run IDs, commit IDs, or checksums.

## Installation and upgrade posture

A clean installation uses the `mcp-runtime` or default v0.6.0 artifact as explicitly selected by the operator:

```bash
ARTIFACT=/absolute/path/to/termux-mcp-server
ARTIFACT_SHA256="$(sha256sum "$ARTIFACT" | awk '{print $1}')"
scripts/termux_deploy.sh install \
  --artifact "$ARTIFACT" \
  --version 0.6.0 \
  --sha256 "$ARTIFACT_SHA256"
```

The configuration layout, fixed `mcp_runtime` service name, versioned deployment root, safe-root policy, authentication posture, and Streamable HTTP protocol are unchanged by the version-only metadata transition.

The physically qualified v0.5.1 exact-main candidate may be retained privately as an internal upgrade/recovery baseline while validating v0.6.0. It is not an authoritative public installation source. There is no previous governed public release for an end user to upgrade from.

## Rollback and removal posture

When v0.6.0 is installed as a clean first release, no `previous` target exists. `rollback` must therefore remain unavailable until a later complete release has been installed as an upgrade. Operators can remove a clean installation while preserving configuration:

```bash
scripts/termux_deploy.sh uninstall
```

During internal qualification, an upgrade from the private validated v0.5.1 baseline must prove failed-candidate recovery, successful v0.6.0 activation, failed-rollback recovery, successful rollback, and configuration-preserving uninstall through the canonical validator.

## Known limitations

- Android platform control, shell fallback, arbitrary command execution, global process inspection, arbitrary service mutation, package/network mutation, and other high-impact actions remain disabled.
- GET `/mcp` returns HTTP 405 because server-initiated SSE and replay/resumption are not implemented.
- Sessions are in-memory, bounded, and intentionally lost on restart.
- Workflow artifacts expire and are not a durable distribution channel.
- No byte-for-byte reproducible-build claim is made.

## Publication stop conditions

Do not create the tag or GitHub Release if any of the following is true:

- final exact-main CI, Security, or any Android posture is missing or failed;
- the five final manifests do not identify the same immutable source commit and version;
- downloaded checksums, sizes, ELF identity, feature posture, or embedded version disagree;
- neither a direct schema-valid `releaseEligible=true` physical report nor a schema-valid `releaseQualificationEligible=true` inherited-observation report exists;
- the native ARM64 official-Termux exact-artifact report is missing or failed;
- observation inheritance reports any runtime, dependency, build-input, deployment, or bridge-digest change;
- a blocking review thread, merge conflict, newer failed run, or release-document inconsistency remains;
- durable asset names/checksums or the GitHub Release body have not been independently rechecked.

Tagging and publishing remain separate maintainer actions after this release-preparation pull request and the final exact-main evidence are complete.
