Those are exactly sixteen governed assets. The staging manifest and its supporting members remain inside the raw tar rather than becoming extra Release assets. GitHub-generated source ZIP/tar downloads are tag-derived conveniences, not governed Android assets, and v0.6.0 makes no separate SBOM asset claim.

The default binary exposes only coarse health/readiness endpoints. The `mcp-runtime` binary additionally exposes the authenticated MCP 2025-11-25 Streamable HTTP surface and documented 17-tool baseline. Battery, volume-status, command, and volume-control artifacts each add only their separately gated surface. The governed `full-suite` artifact composes them all but remains at exactly 17 tools with every optional runtime flag off and reaches exactly 21 only when all four flags are enabled independently. Filesystem and volume mutations keep separate default-disabled runtime gates and exact-operation grants. Artifact names and manifests must keep all seven postures unambiguous; raw Cargo `--all-features` is only a development compatibility lane and is not a durable release asset.

Final evidence must be generated after the last v0.6.0 qualification change reaches `main`. It must reconcile one immutable SHA across CI, Security, seven Android workflow manifests, downloaded binary digests, embedded `termux-mcp-server 0.6.0` output, aggregate schema/gate-v3 evidence, and native ARM64 official-Termux runtime validation. Canonical validator v11 must emit direct evidence schema v2 bound to the exact downloaded workflow full-suite digest. Device harness v11 must produce a fresh physical observation of the same commit and record its separate locked on-device native-build digest; byte equality across those different build paths is not claimed. The committed v0.5.1 report and historical bridge cannot qualify this changed feature/artifact surface. Values are recorded in the GitHub Release body only after they exist; this repository does not contain guessed run IDs, commit IDs, or checksums.

The qualifying Android run retains all seven bundles and native-emulation evidence for 30 days and is never regenerated by pushing a version tag. After the direct physical pass, `package_physical_qualification.sh` emits a sanitized closed envelope that binds the validator report digest, private raw harness report digest, workflow full-suite digest, and separate native full-suite digest. The protected staging lane in [`PUBLIC_RELEASE.md`](PUBLIC_RELEASE.md) then copies the exact qualified workflow bytes into the seven final filenames and a deterministic tar. That public-repository Actions artifact is not confidential and remains `staged_not_released`; it neither creates nor authorizes a tag or GitHub Release.

Publication is a later independent state machine. An authorized maintainer creates one protected annotated tag and empty exact-tag draft outside the workflow. The `release-production` environment approves attachment of the fixed sixteen assets; a fresh read-only runner re-downloads and proves every draft byte; the disjoint `release-final` environment separately approves publication; and public completion requires `immutable: true` plus public re-download proof. A tag or populated draft is still not an installation source.

## Installation and upgrade posture

A clean installation uses the least-privilege artifact that meets the deployment need, or the governed `full-suite` v0.6.0 artifact when all optional providers are required, as explicitly selected by the operator:

```bash
ARTIFACT=/absolute/path/to/termux-mcp-server
ARTIFACT_SHA256="$(sha256sum "$ARTIFACT" | awk '{print $1}')"
scripts/termux_deploy.sh install \
  --artifact "$ARTIFACT" \
  --version 0.6.0 \
  --sha256 "$ARTIFACT_SHA256"
```

The configuration layout, fixed `mcp_runtime` service name, versioned deployment root, safe-root policy, authentication posture, and Streamable HTTP protocol are unchanged by the version-only metadata transition.

The physically qualified v0.5.1 exact-main candidate may be retained privately as an internal upgrade/recovery baseline while validating v0.6.0. It is not an authoritative public installation source. There is no previous governed public release for an end user to upgrade from.

## Rollback and removal posture

When v0.6.0 is installed as a clean first release, no `previous` target exists. `rollback` must therefore remain unavailable until a later complete release has been installed as an upgrade. Operators can remove a clean installation while preserving configuration:

```bash
scripts/termux_deploy.sh uninstall
```

During internal qualification, an upgrade from the private validated v0.5.1 baseline must prove failed-candidate recovery, successful v0.6.0 activation, failed-rollback recovery, successful rollback, and configuration-preserving uninstall through the canonical validator.

## Known limitations

- Android control beyond exact-stream volume, shell fallback, arbitrary command execution, global process inspection, arbitrary service mutation, package/network mutation, and unrelated high-impact actions remain disabled.
- GET `/mcp` returns HTTP 405 in the default JSON posture; the separately configured SSE posture is finite, replay-bounded, and never broadcasts across streams.
- Sessions are in-memory, bounded, and intentionally lost on restart.
- Directory, file-copy, file-trash, file-write, and volume authorities with the same exact family/key-id/key/principal share replay, clock, and capacity state across independently constructed routers in one process. Grants expire after 60 seconds and lose their process-local replay state on restart; distinct capability codes prevent cross-use, immediate invalidation requires key rotation, and multi-process consumers require an external atomic one-use coordinator.
- Workflow artifacts are retained for 30 days, then expire; they are qualification inputs rather than a durable distribution channel.
- No byte-for-byte reproducible-build claim is made.

## Publication stop conditions

Do not create the protected annotated tag or empty draft until the exact-main qualification and stage are complete. Do not attach or publish if any of the following is true:

- final exact-main CI, Security, or any Android posture is missing or failed;
- the seven final manifests do not identify the same immutable source commit and version;
- downloaded checksums, sizes, ELF identity, feature posture, or embedded version disagree;
- a fresh direct validator-v11/schema-v2 physical report bound to the full-suite digest does not say `releaseEligible:true`;
- the native ARM64 official-Termux exact-artifact report is missing or failed;
- aggregate schema/gate-v3 evidence does not bind the exact full-suite digest/manifest and prove the 17-disabled, four isolated 18-tool, 21-enabled, and four-filesystem-denial contracts;
- the physical-qualification envelope does not bind both evidence-report digests and the distinct workflow/native full-suite lineages without exposing raw device metadata;
- the protected release stage is absent, non-deterministic, built from anything other than the exact qualified bytes, or lacks a closed `staged_not_released` manifest;
- the `release-qualification` environment protections have not been independently verified before dispatch;
- release immutability is not enabled or cannot be confirmed by both separate environment-only Administration-read policy credentials;
- the active `v*` ruleset permits tag movement/deletion or GitHub Actions bypass, or the exact version tag is not annotated and bound to the staged commit;
- `release-production` and `release-final` do not have disjoint reviewers, prevent-self-review, main-only policy, disabled administrator bypass, and their exact environment-only guards;
- the exact-tag draft was not pre-created with the exact version title, a blank body, `draft:true`, `prerelease:false`, and zero assets;
- the protected attachment does not produce exactly seven binaries, seven sidecars, `SHA256SUMS`, and the unchanged raw staging tar;
- fresh read-only draft verification, post-final-approval revalidation, `immutable:true`, or public sixteen-asset re-download proof is missing or mismatched;
- a blocking review thread, merge conflict, newer failed run, or release-document inconsistency remains;
- durable asset names/checksums or the GitHub Release body have not been independently rechecked.

A v0.6.0 GitHub Release is authorized and installable only if tagging, empty-draft creation, protected attachment, final publication approval, and immutable public proof all pass against the final exact-main evidence.
